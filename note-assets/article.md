# AI同士にペアプログラミングさせたら、コードの品質が明らかに上がった話

<!-- 画像: 01-title.svg -->

## はじめに

Claude Code を日常的に使ってコードを書いている。便利なのだが、1つ気になることがあった。

**AIが書いたコードを、結局人間がレビューしないといけない。**

AIにコードを書かせて、自分でレビューして、修正を指示して、また確認して……。これ、ペアプロの片方が自分である限り、認知負荷はあまり減っていない。

そこで考えた。**レビューもAIにやらせればいいのでは？**

Claude Code（Anthropic）と OpenAI Codex、2つのAIを組み合わせて自動でペアプログラミングさせるツール `pair` を作った。

<!-- 画像: 02-why.svg -->

## なぜ「異なるAI」を組み合わせるのか

同じモデルに書かせて同じモデルにレビューさせても、盲点は盲点のままだ。

Claude と Codex はアーキテクチャも学習データも異なる。Claude は深い推論と設計判断に強く、Codex は実行ベースの検証に強い。**モデルの多様性**が、単一モデルでは見つけられないバグや設計ミスを炙り出す。

人間のペアプログラミングで「2人の視点があると品質が上がる」のと同じ原理だ。

## 2つの方式

このツールには大きく分けて2つの使い方がある。

### 1. リレー方式（`pair`）

<!-- 画像: 03-architecture.svg -->

```bash
pair "認証機能を実装して"
```

これだけで以下が自動的に行われる：

1. **Claude** がコードを実装する
2. Claude が「ハンドオフメモ」を書く（何をやったか、なぜその判断をしたか、残課題は何か）
3. **Codex** が diff とハンドオフメモを読み、レビュー・改善する
4. Codex もハンドオフメモを書く
5. 1に戻る（指定ラウンド数だけ繰り返す）

各ラウンドの後に自動コミットされるので、途中経過も追える。

#### ハンドオフメモという仕組み

人間のペアプロでも「何を考えてこう書いたか」が伝わらないと、レビュアーは余計な書き直しをしてしまう。AI同士でもこれは同じだった。

そこで各ターンの終わりに、必ず `.pair-handoff.md` に引き継ぎメモを書かせるようにした：

```markdown
## やったこと
- JWT認証ミドルウェアを追加
- トークン検証のユーティリティ関数を作成

## 判断の理由
- sessionよりJWTを選択（ステートレスにしたいため）
- bcryptではなくargon2を採用（メモリハードで安全）

## 次のパートナーへ
- リフレッシュトークンが未実装
- エラーレスポンスの形式を統一したい
```

**diff だけでなく「意図」が伝わる**ことで、良い判断はそのまま維持され、本当に問題のある部分だけが修正される。これがかなり効く。

#### リアルタイム進捗表示

実行中は何が起きているかリアルタイムで見える：

```
[Claude] 開始...
  [5s]  ▶ Read
      → page.tsx
  [12s] ▶ Edit
      → account-card.tsx
  [30s] ▶ Bash
      $ bun run lint
  ✓ 完了 1m45s | 8ターン | $0.42

[Codex] 開始...
  [3s]  ▶ shell
  ✓ 完了 52s
```

ツール呼び出し、対象ファイル、実行時間、コストまで表示される。放置していても「今どこまで進んだか」がわかる。

### 2. ナビゲーター方式（`pair-nav`）

<!-- 画像: 04-navigator.svg -->

```bash
pair-nav start
```

これを実行すると、Claude Code の hooks が設定される。あとは**普通に Claude と対話するだけ**。

裏で何が起きるか：

1. あなたが Claude に指示を出す
2. Claude がコードを書く
3. Claude の応答が完了した瞬間、**Stop hook** が発火
4. Codex が差分を自動でレビュー
5. 次にあなたがプロンプトを送ると、**UserPromptSubmit hook** が発火
6. Codex のフィードバックが Claude のコンテキストに自動注入される
7. Claude がフィードバックを見て改善する

問題なければ Codex は「LGTM」を返してスキップ。差分が小さすぎる場合もスキップ。**ノイズにならないよう設計**している。

リレー方式との使い分け：
- **リレー**：まとまったタスクを丸投げしたいとき
- **ナビゲーター**：自分で Claude と対話しながら、裏でセカンドオピニオンが欲しいとき

## 4つのペアプロモード

<!-- 画像: 05-modes.svg -->

リレー方式には4つのモードがある：

| モード | 動き | 向いている場面 |
|--------|------|---------------|
| **ping-pong**（デフォルト） | Claude → Codex → Claude → ... | ゼロからの機能実装 |
| **review** | 片方が実装、もう片方がレビュー＆修正 | 既存コードの改善 |
| **implement-review** | Claude実装 → Codex がレビュー文書作成 → Claude が修正 | レビュー記録を残したいとき |
| **parallel** | 両方が同時に実装 → worktree 分離 → マージ | とにかく速く |

特に面白いのは **parallel モード**。git worktree を使って Claude と Codex がそれぞれ独立したブランチで同時に実装し、最後にマージする。コンフリクトが起きたら Claude が両方のハンドオフメモを読んで自動解決する。

## 苦労したところ：macOS での bash 互換性

開発中に地味に苦労したのが **macOS の bash 3.2 との互換性**だ。

macOS にプリインストールされている bash は GPLv2 の 3.2 で止まっている。`${var,,}`（小文字変換）のような bash 4 以降の機能は使えない。

さらに厄介だったのが**パイプラインのタイムアウト処理**。`timeout` コマンドは macOS にはデフォルトで入っていないし、パイプの途中のプロセスだけ kill してもパイプライン全体が止まらない。

最終的に、**サブシェル + プロセスグループ kill** というパターンに落ち着いた：

```bash
(
  claude ... | tee "$log" | while read -r line; do ...; done
) &
local pid=$!

# プロセスグループごと kill
(
  sleep "$TIMEOUT"
  kill -- -"$pid" 2>/dev/null
) &
```

サブシェル全体をバックグラウンドで起動し、`kill -- -$pid` でプロセスグループごと停止する。これで macOS でもパイプラインがハングしなくなった。

ちなみに `--timeout 0` と指定すると `sleep 0` が即座に返って即タイムアウトするというバグも踏んだ。0 は「無制限」の意味で使いたいので、タイムアウト値が 0 ならウォッチャーを起動しないように修正した。

## Codex のクラッシュ対策

Codex は `--json` フラグ付きで実行するとストリーミング JSON で進捗が取れるのだが、稀にクラッシュする（exit code 128+ のシグナル系エラー）。

そこで**リトライ + フォールバック**を実装した：

1. 初回は `--json` 付きで実行（進捗表示あり）
2. クラッシュしたら `--json` なしで再試行（プレーンテキストモード）
3. それでもダメなら諦める

これで安定性がかなり上がった。

## セットアップ

<!-- 画像: 06-setup.svg -->

```bash
git clone https://github.com/gokkoclub/pair.git /tmp/pair && /tmp/pair/setup.sh
```

1行でインストール完了。CLI（`pair`, `pair-nav`）と Claude Code のスラッシュコマンド（`/pair`, `/pair-nav`）が両方使えるようになる。

### 前提条件

- `claude` CLI（[Claude Code](https://claude.com/claude-code)）
- `npx @openai/codex`（OpenAI API キーが設定済みであること）
- git リポジトリ内で実行すること

### すぐ試せる例

```bash
# リレー方式（3ラウンド）
pair "RESTful APIエンドポイントを追加して"

# 5ラウンドで徹底的に
pair --rounds 5 "認証機能を実装"

# Codex に先に書かせる
pair --claude-first false "テストを追加"

# ナビゲーター方式
pair-nav start
# → Claude Code を再起動して普通に使う
```

## まとめ

「AI にコードを書かせる」時代から「**AI 同士にペアプロさせる**」時代へ。

実際に使ってみると、単一 AI に書かせたときと比べて：
- バグの見落としが減る（異なるモデルの視点）
- 設計判断が改善される（ハンドオフメモで意図が伝わる）
- 人間の認知負荷が激減する（結果だけ確認すればいい）

もちろん最終的な判断は人間がすべきだが、**AI が書いて AI がレビューした後のコード**を確認するのと、**AI が書いたコードを全部自分でレビューする**のとでは、負荷がまったく違う。
