# AI同士にペアプログラミングさせたら、コードの品質が明らかに上がった話

[画像: 01-title.png]

---

## きっかけ

Claude Code を日常的に使ってコードを書いている。便利だが、1つ気になることがあった。

AIが書いたコードを、結局人間がレビューしないといけない。

AIにコードを書かせて、自分でレビューして、修正を指示して、また確認して……。ペアプロの片方が自分である限り、認知負荷はあまり減っていない。

そこで考えた。レビューもAIにやらせればいいのでは？

Claude Code（Anthropic）と OpenAI Codex、2つのAIを組み合わせて自動でペアプログラミングさせるツール「pair」を作った。

[画像: 02-why.png]

---

## なぜ「異なるAI」を組み合わせるのか

同じモデルに書かせて同じモデルにレビューさせても、盲点は盲点のままだ。

Claude と Codex はアーキテクチャも学習データも異なる。Claude は深い推論と設計判断に強く、Codex は実行ベースの検証に強い。モデルの多様性が、単一モデルでは見つけられないバグや設計ミスを炙り出す。

人間のペアプロで「2人の視点があると品質が上がる」のと同じ原理だ。

## リレー方式

[画像: 03-architecture.png]

```
pair "認証機能を実装して"
```

これだけで以下が自動的に行われる：

1. Claude がコードを実装する
2. Claude が「ハンドオフメモ」を書く（何をやったか、なぜその判断をしたか、残課題は何か）
3. Codex が diff とハンドオフメモを読み、レビュー・改善する
4. Codex もハンドオフメモを書く
5. 1に戻る（指定ラウンド数だけ繰り返す）

ポイントはハンドオフメモ。diff だけでなく「意図」が伝わることで、良い判断はそのまま維持され、本当に問題のある部分だけが修正される。

リアルタイムで進捗も見える：

```
[Claude] 開始...
  [5s]  ▶ Read → page.tsx
  [12s] ▶ Edit → auth.ts
  [30s] ▶ Bash $ bun run lint
  ✓ 完了 1m45s | 8ターン | $0.42

[Codex] 開始...
  [3s]  ▶ shell
  ✓ 完了 52s
```

## ナビゲーター方式

[画像: 04-navigator.png]

```
pair-nav start
```

これを実行すると、Claude Code の hooks が設定される。あとは普通に Claude と対話するだけ。

裏で何が起きるか：
1. Claude がコードを書く
2. Claude の応答完了 → Codex が差分を自動レビュー
3. 次のプロンプトで Codex のフィードバックが自動注入される
4. Claude がフィードバックを見て改善する

問題なければ Codex は LGTM を返してスキップ。ノイズにならないよう設計している。

## 4つのモード

[画像: 05-modes.png]

| モード | 動き | 場面 |
|--------|------|------|
| ping-pong | Claude → Codex → Claude → ... | ゼロからの機能実装 |
| review | 片方が実装、もう片方がレビュー＆修正 | 既存コードの改善 |
| implement-review | Claude実装 → Codexレビュー → Claude修正 | レビュー記録を残したいとき |
| parallel | 両方が同時に実装 → worktree → マージ | とにかく速く |

parallel モードは特に面白い。git worktree で Claude と Codex がそれぞれ独立ブランチで同時実装し、最後にマージ。コンフリクトは Claude が両方のハンドオフメモを読んで自動解決する。

## 苦労話：macOS の bash 互換性

macOS の bash は 3.2 で止まっている。`${var,,}` のような bash 4+ の機能は使えない。

さらにパイプラインのタイムアウト処理が厄介で、パイプの途中のプロセスだけ kill してもパイプライン全体が止まらない。

最終的にサブシェル + プロセスグループ kill で解決した：

```bash
(
  claude ... | tee "$log" | while read -r line; do ...; done
) &
local pid=$!
(
  sleep "$TIMEOUT"
  kill -- -"$pid" 2>/dev/null
) &
```

`--timeout 0` で `sleep 0` が即返って即タイムアウトするバグも踏んだ。0 = 無制限にするため、TIMEOUT > 0 のときだけウォッチャーを起動するよう修正。

## まとめ

「AI にコードを書かせる」時代から「AI 同士にペアプロさせる」時代へ。

単一 AI と比べて：
- バグの見落としが減る（異なるモデルの視点）
- 設計判断が改善される（ハンドオフメモで意図が伝わる）
- 人間の認知負荷が激減する（結果だけ確認すればいい）

AI が書いて AI がレビューした後のコードを確認するのと、AI が書いたコードを全部自分でレビューするのとでは、負荷がまったく違う。
